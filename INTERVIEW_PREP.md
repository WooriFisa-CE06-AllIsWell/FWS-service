# FWS 면접 준비 — 예상 질문 & 답변

---

## 프로젝트 전반

**Q. 이 프로젝트를 왜 만들었나요?**

클라우드 환경에서 개발자나 학습자가 VM이 필요할 때마다 vCenter에 직접 접속해서 수동으로 생성하는 과정이 번거롭습니다. 웹 UI에서 OS와 스펙을 선택하면 VM이 자동으로 생성되고, 별도 SSH 클라이언트 없이 브라우저에서 바로 접속할 수 있도록 셀프서비스 형태로 구현했습니다.

---

**Q. 전체 아키텍처를 설명해주세요.**

프론트엔드(nginx), 백엔드(FastAPI), VM 인프라(vSphere), 웹 터미널(Guacamole) 4개 컴포넌트로 구성됩니다. 핵심은 Service-Server의 NIC 이중화입니다. 사용자망(192.168.x.x)과 vCenter 내부망(172.18.x.x)에 각각 NIC을 붙여, 백엔드가 외부 요청도 받고 vCenter/Guacamole에도 접근할 수 있도록 했습니다.

---

## 기술 선택

**Q. FastAPI를 선택한 이유가 있나요?**

Python 기반의 비동기 웹 프레임워크이고, Swagger UI가 자동 생성되어 API 테스트와 문서화가 동시에 됩니다. vSphere 연동에 쓰는 pyVmomi도 Python 라이브러리라 스택을 통일할 수 있었습니다.

---

**Q. vCenter REST API를 쓰지 않고 왜 pyVmomi(SOAP)를 사용했나요?**

처음에는 REST API로 구현했는데, 실제 환경(vSphere 7.0.3)에서 `POST /api/vcenter/vm?action=clone`이 404로 떨어졌습니다. 확인해보니 해당 엔드포인트는 vSphere 7.0.2U2 이상에서만 지원되는 기능이었습니다. pyVmomi는 구버전 vSphere에서도 안정적으로 동작하는 VMware 공식 Python 라이브러리여서 VM 생성/삭제는 SOAP API로, 자원 조회는 REST API로 나눠서 처리하는 하이브리드 구조를 채택했습니다.

---

**Q. Guacamole를 선택한 이유는 무엇인가요?**

브라우저에서 SSH 접속을 제공하는 오픈소스 중 Apache Guacamole가 REST API를 통해 연결을 동적으로 등록/삭제할 수 있습니다. VM 생성 시 자동 등록, 반납 시 자동 삭제가 필요했기 때문에 REST API 지원 여부가 핵심 선택 기준이었습니다.

---

## 트러블슈팅

**Q. 가장 어려웠던 문제가 무엇인가요?**

Guacamole REST API 연동 과정이 가장 복잡했습니다. 처음에는 파일 기반 인증(`user-mapping.xml`)으로 설정되어 있어서 REST API로 연결을 생성하면 PERMISSION_DENIED가 계속 떴습니다. 인증 방식 자체가 문제라는 걸 파악하는 데 시간이 걸렸고, docker-compose에 MySQL 컨테이너를 추가하고 DB 기반 인증으로 전환해서 해결했습니다.

---

**Q. vSphere REST API에서 필터 파라미터 오류는 어떻게 해결했나요?**

`filter.power_states=POWERED_ON` 같은 쿼리 파라미터를 붙이면 400 오류가 났습니다. vSphere 7.0.3 버전에서 일부 필터가 미지원이었는데, 전체 VM 목록을 먼저 가져온 뒤 Python 코드에서 조건에 맞는 항목만 필터링하는 방식으로 우회했습니다. API 스펙과 실제 환경의 차이를 직접 확인하고 대응하는 경험이었습니다.

---

**Q. 네트워크 구성에서 고려한 점은 무엇인가요?**

사용자 PC에서 브라우저로 접근하는 망(192.168.x.x)과 vCenter/VM이 있는 내부망(172.18.x.x)이 분리되어 있었습니다. 처음에는 사용자 PC에서 직접 백엔드를 실행했는데, Guacamole 서버(192.168.0.38)에 접근이 안 됐습니다. 두 망 모두에 NIC이 연결된 Service-Server를 배포 대상으로 정하고, 거기서 백엔드를 실행함으로써 해결했습니다.

---

## 코드/설계

**Q. 환경변수를 `.env`로 분리한 이유는 무엇인가요?**

vCenter 비밀번호, Guacamole 계정 등 민감한 정보를 코드에 하드코딩하면 Git에 올라갈 위험이 있습니다. `.env`는 `.gitignore`에 등록해서 저장소에 포함되지 않게 하고, `.env.example`로 팀원들이 어떤 값을 채워야 하는지 알 수 있도록 했습니다.

---

**Q. MOCK_MODE를 넣은 이유는 무엇인가요?**

실제 vCenter 환경 없이도 프론트엔드와 API 인터페이스를 독립적으로 개발하고 테스트할 수 있게 하기 위해서입니다. `MOCK_MODE=true`로 설정하면 더미 데이터를 반환하기 때문에 인프라 접근 없이도 전체 UI 흐름을 확인할 수 있습니다.

---

**Q. VM 생성 후 IP를 어떻게 가져오나요?**

VM 클론 후 파워온까지는 금방 되지만, 게스트 OS가 부팅되고 VMware Tools를 통해 IP가 보고될 때까지 시간이 걸립니다. 5초 간격으로 VM의 `guest.ipAddress`를 폴링하면서 최대 5분까지 대기하는 로직을 구현했습니다. 타임아웃이 지나면 예외를 발생시키고 에러를 반환합니다.

---

## 개선 가능성 (추가 질문 대비)

**Q. 개선하고 싶은 부분이 있나요?**

몇 가지 있습니다:

1. **인증/권한 없음**: 현재는 누구나 VM을 생성/삭제할 수 있습니다. 로그인 기능과 사용자별 VM 소유권 관리가 필요합니다.
2. **VM 생성 시간**: 클론 완료까지 3~5분이 걸립니다. 웹소켓이나 SSE(Server-Sent Events)로 진행 상황을 실시간으로 보여주면 UX가 개선될 것입니다.
3. **VM 목록 조회 없음**: 현재 생성한 VM 이름을 직접 입력해야 반납할 수 있습니다. 본인이 생성한 VM 목록을 조회하는 기능이 있으면 더 편리합니다.
4. **Guacamole 비밀번호 노출**: SSH 연결에 비밀번호를 사용하고 있는데, 키 기반 인증으로 전환하면 보안이 강화됩니다.
